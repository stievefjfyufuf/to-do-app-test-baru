# Database and API Design — To-Do App Mahasiswa

Status: Design baseline untuk UI design dan issue planning
Tanggal: 29 Juni 2026
Input: `03-specification.md` (`DOC-003`), `05-validation-change.md` (`DOC-005`), dan `06-architecture-design.md` (`DOC-006`)
Scope: Requirement `Ready-MVP` sesuai `DEC-023` dan `DEC-024`

## 1. Design Principles

- PostgreSQL adalah authoritative server store (`ADR-004`, `COMP-010`); IndexedDB adalah device-local replica/cache (`ADR-002`, `COMP-002`).
- Semua server data pribadi memiliki ownership `user_id`; API tidak menerima ownership dari client sebagai sumber otorisasi (`ADR-007`, NFR-006).
- UUID dibuat client untuk task/command agar create dapat dilakukan offline tanpa remapping ID (`REQ-017`, `REQ-018`).
- Semua mutasi masuk melalui application command pipeline yang sama, baik endpoint langsung maupun batch sync (`COMP-003`, `COMP-007`).
- Optimistic concurrency memakai `baseVersion`; retry memakai `commandId`/`Idempotency-Key` (`ADR-005`, AC-020/021).
- Soft-delete task dan account-deletion grace memakai 30 hari (`ADR-006`, REQ-005/022/024).
- Deferred requirements tidak memiliki tabel atau endpoint MVP khusus.

## 3. PostgreSQL Tables

### DATA-001 — `users`

Purpose: identity root dan account-deletion lifecycle (`REQ-015`, `REQ-024`, `COMP-005/009`).

| Field | Type | Null | Rules |
|---|---|:---:|---|
| `id` | UUID | No | PK; generated by server |
| `email_normalized` | VARCHAR(254) | No | lowercase/trimmed; unique |
| `password_hash` | TEXT | No | adaptive password hash; never returned |
| `status` | VARCHAR(24) | No | CHECK `active`, `pending_deletion`; default `active` |
| `scheduled_deletion_at` | TIMESTAMPTZ | Yes | required iff status `pending_deletion`; max baseline `now + 30 days` |
| `created_at` | TIMESTAMPTZ | No | default `now()` |
| `updated_at` | TIMESTAMPTZ | No | application-managed |

Constraints/indexes: unique `email_normalized`; index `scheduled_deletion_at` WHERE status=`pending_deletion`. Account deletion cascades to private module data after the grace period.

### DATA-002 — `user_profiles`

### DATA-005 — `tasks`

Purpose: authoritative task aggregate (`REQ-001–REQ-008`, `REQ-013`, `REQ-022`, `COMP-006`).

| Field | Type | Null | Rules |
|---|---|:---:|---|
| `id` | UUID | No | PK; client-generated allowed |
| `user_id` | UUID | No | FK users ON DELETE CASCADE |
| `title` | VARCHAR(200) | No | trimmed; at least 1 non-space char |
| `description` | VARCHAR(5000) | Yes | null or trimmed content |
| `status` | VARCHAR(16) | No | CHECK `active`, `completed`; default `active` |
| `priority` | VARCHAR(16) | Yes | CHECK null, `low`, `medium`, `high` |
| `due_at` | TIMESTAMPTZ | Yes | canonical UTC instant |
| `due_timezone` | VARCHAR(64) | Yes | valid IANA zone; required with due_at |
| `due_has_explicit_time` | BOOLEAN | No | false means local time defaulted to 23:59 |
| `completed_at` | TIMESTAMPTZ | Yes | non-null iff status=`completed` |
| `version` | BIGINT | No | CHECK version >=1; increments each accepted mutation |
| `deleted_at` | TIMESTAMPTZ | Yes | non-null means trash; purge eligible at +30 days |
| `created_at` | TIMESTAMPTZ | No | supplied offline timestamp validated; server fallback |
| `updated_at` | TIMESTAMPTZ | No | server commit time |

Constraints: due fields are both null or both present; completed status matches `completed_at`; client timestamp cannot determine authorization/version ordering. Indexes: `(user_id, deleted_at, status, updated_at DESC, id)`, `(user_id, deleted_at, due_at, id)`, `(user_id, deleted_at, priority, id)`, and partial `(deleted_at)` WHERE deleted_at IS NOT NULL.

|---|---|:---:|---|

| `user_id` | UUID | No | FK users ON DELETE CASCADE |
| `device_id` | UUID | No | FK devices ON DELETE CASCADE |
| `operation` | VARCHAR(40) | No | allowlisted task operation |

## 5. Relationship, Integrity, and Retention Rules

- User-to-task/device/session is 1:N; user-to-profile is 1:1; task-to-completion-event is 1:N.
- All resource queries include authenticated `user_id`; cross-account lookup returns `404 NOT_FOUND` to avoid disclosing existence (AC-017).
- Accepted task mutation, completion event, sync change, and idempotency response commit in one database transaction.
- `version` is server-controlled and increments once per accepted mutation, including complete/reopen/trash/restore.
- Permanent task deletion cascades completion events; statistics no longer count permanently deleted tasks.
- Trash purge: `deleted_at < now() - interval '30 days'`, processed idempotently in bounded batches.
- Account deletion job verifies `scheduled_deletion_at <= now()` and current status before cascade; cancellation makes queued deletion a no-op.
- Sync change payload retention is 90 days. Snapshot resets are explicit and never discard local pending commands.
- Idempotency records are retained 90 days. Commands older than the supported replay window return `COMMAND_EXPIRED` for manual recovery rather than being applied ambiguously.

## 6. API Conventions

- Base path: `/api/v1`; content type `application/json; charset=utf-8`.
- Authentication: opaque session cookie `HttpOnly`, `Secure`, `SameSite=Lax`; state-changing requests also require CSRF token/header.
- Resource authorization always uses session `user_id`; request body cannot set `userId`.
- IDs are lowercase UUID strings; timestamps are RFC 3339 UTC; timezone is IANA name.
- Direct mutation requests require `Idempotency-Key: <uuid>`; existing-resource mutation also requires `If-Match: "<version>"`.
- `X-Client-Instance-Id` identifies the registered device but never grants authority.
- Success envelope: `{ "data": ..., "meta": { ... } }`; error envelope is defined in section 10.
- List pagination is opaque cursor based; default limit 25, maximum 100. Sync change limit maximum 200.
- API evolution is additive within v1. Unknown response fields must be ignored by clients.

### 8.3 Device and Synchronization

| ID | Method / Path | Purpose | Auth | Request / Validation | Response / Errors | Traceability |
|---|---|---|---|---|---|---|
| API-019 | `POST /devices/register` | Register/update PWA instance | Session | clientInstanceId, optional displayName; idempotency | 200 device ID and cursor policy | REQ-017/018; COMP-003 |
| API-020 | `POST /sync/commands` | Replay ordered offline command batch | Session + device | commands max 50/1 MiB | 200 per-command results; conflicts are explicit | REQ-017/018; AC-019–021 |
| API-021 | `GET /sync/changes` | Pull changes after cursor | Session + device | cursor, limit <=200 | 200 changes + nextCursor/hasMore; 409 SYNC_RESET_REQUIRED | REQ-018; AC-020; NFR-005 |
| API-022 | `GET /sync/snapshot` | Rebuild replica after cursor expiry | Session + device | cursor pagination; no pending command deletion | 200 canonical tasks/tombstones + snapshotCursor | REQ-008/018; NFR-003/005 |
| API-023 | `GET /sync/conflicts` | List open/resolved conflicts | Session + device | status, cursor, limit | 200 conflict summaries | REQ-018; AC-021 |
| API-024 | `GET /sync/conflicts/{conflictId}` | Read both retained versions | Session | owned conflict | 200 Conflict; 404 cross-user | REQ-018; AC-021; NFR-006 |
| API-025 | `POST /sync/conflicts/{conflictId}/resolve` | Resolve with local/server/merged fields | Session | resolution, merged payload if needed, latest serverVersion, idempotency | 200 canonical Task; 409 if server advanced again | REQ-018; AC-021 |

`POST /sync/commands` processes commands sequentially per submitted order but commits each command independently. A failed/conflicting command does not silently discard later commands; results identify `applied`, `replayed`, `conflict`, `retryable_failure`, or `permanent_failure`.

Snapshot replacement treats the returned snapshot as authoritative only for server-synced rows at `snapshotCursor`. Local rows with pending commands remain isolated until replay/resolution; they are never removed merely because they are absent from the snapshot.

| API-027 | `POST /account-deletion` | Schedule deletion +30 days | Session + recent reauth | password/reauth token, confirmation, idempotency | 202 scheduledDeletionAt; existing schedule replays | REQ-024; AC-027 |
| API-028 | `DELETE /account-deletion` | Cancel before deadline | Session | idempotency | 204; 410 if irreversible job has begun | REQ-024; AC-027 |

An account remains usable during the grace period and displays a persistent pending-deletion state. The deletion worker re-checks status/deadline inside its transaction before deleting.

## 9. Authorization, Validation, and Rate Limits

| Area | Rule |
|---|---|
| Authentication | Only session hash is stored; raw cookie/token never enters DB/logs. Expired/revoked sessions return 401. |
| Authorization | Repository/service receives server-derived user_id. Cross-account task/conflict IDs return 404. |
| CSRF | Required for cookie-authenticated POST/PATCH/DELETE except public register/login/reset endpoints, which use origin checks and rate limits. |
| Password | Minimum 12 chars; reject known-compromised/common passwords where service/library is available; hash parameters versioned. |
| Title/description | Trim title and reject blank; max 200/5,000; output encoded by UI. |
| Due/timezone | Valid local datetime and IANA zone; DST-invalid local times return validation error; missing time becomes 23:59. |
| Version | If-Match/baseVersion required for existing task mutation; stale version creates/returns conflict, never last-write-wins. |
| Idempotency | UUID required for every mutation; key/payload mismatch returns 409. |
| Pagination | Opaque signed/encoded cursor; invalid/expired cursor returns 422 except sync cursor, which may require snapshot. |
| Public auth limits | Login 5/min/account+IP bucket; reset request 3/hour/email+IP bucket; responses do not reveal account existence. |
| Authenticated limits | General 120/min/session; sync command batches 12/min/device; provider tuning may lower/raise with evidence. |

## 10. Error Responses

```json
{
  "error": {
    "code": "VERSION_CONFLICT",
    "message": "The task changed on another device.",
    "correlationId": "uuid",
    "fieldErrors": [],
    "retryable": false,
    "details": {
      "conflictId": "uuid",
      "serverVersion": 4
    }
  }
}
```

| HTTP | Stable Code | Meaning/Client Action |
|---:|---|---|
| 400 | `MALFORMED_REQUEST` | Invalid JSON/content type; do not retry unchanged. |
| 401 | `AUTH_REQUIRED` / `INVALID_CREDENTIALS` | Login/reauth required; public auth response stays generic. |
| 403 | `CSRF_FAILED` / `ACTION_FORBIDDEN` | Security rule denied; private resource ownership still uses 404. |
| 404 | `NOT_FOUND` | Resource absent or not owned; no existence disclosure. |
| 409 | `VERSION_CONFLICT` | Show/refresh conflict; never overwrite. |
| 409 | `IDEMPOTENCY_KEY_REUSED` | Generate a new key only for a genuinely new command. |
| 409 | `SYNC_RESET_REQUIRED` | Download snapshot while preserving local pending commands. |
| 410 | `RETENTION_EXPIRED` / `COMMAND_EXPIRED` | Restore/replay no longer automatic; show recovery guidance. |
| 413 | `PAYLOAD_TOO_LARGE` | Reduce command batch/request. |
| 422 | `VALIDATION_ERROR` | Display field errors; no retry unchanged. |
| 429 | `RATE_LIMITED` | Honor `Retry-After`. |
| 500 | `INTERNAL_ERROR` | Safe generic message; log correlation only. |
| 503 | `SERVICE_UNAVAILABLE` | Retry with backoff; offline queue remains pending. |

## 11. Transaction and Concurrency Boundaries

- Task command transaction: lock/read task version, validate ownership/version, mutate task, append completion event when needed, append sync change, finalize idempotency response, then commit.
- Create command: reserve `(user_id, task_id)` and command ID atomically; UUID collision with another owned entity returns 409.
- Concurrent duplicate command: unique command ID elects one processor; others wait/read stored response. Stale `processing` rows are recovered by bounded lease logic.
- Conflict path: command finalizes as `conflict` and conflict record is committed atomically without mutating task.
- Conflict resolution is a new command against latest server version; if that version changed again, a new conflict is produced.
- Background purge/account deletion uses `FOR UPDATE SKIP LOCKED` or equivalent leasing and idempotency keys.
- API never promises distributed exactly-once delivery; it guarantees idempotent observable outcomes within the retention window.

## 12. Migration Plan

| Migration | Scope | Compatibility/Rollback |
|---|---|---|
| MIG-001 | users, profiles | Additive initial schema; rollback only before production data. |
| MIG-002 | sessions, reset tokens | Independent identity tables; revoke tokens before destructive rollback. |
| MIG-003 | tasks, completion events, task indexes | Backfill/version defaults before NOT NULL; validate constraints before enforcement. |
| MIG-004 | devices, idempotency commands, sync changes/conflicts | Deploy server read compatibility before client sync activation; offline writes behind capability flag. |
| MIG-005 | background jobs, audit events | Additive; worker deploys after tables and indexes exist. |
| MIG-006 | retention jobs/index tuning | Enable purge only after dry-run metrics and backup verification. |

Rules:

- Prefer expand → migrate/backfill → verify → contract. Do not combine destructive column removal with the release that stops writing it.
- Database migration and API deployment are backward compatible with at least the immediately previous PWA version.
- Index creation on populated large tables uses online/concurrent strategy where supported.
- No production user seed data. Test fixtures use synthetic accounts and deterministic task datasets.
- Before risky migration, verify backup and restore evidence; application rollback must not depend on reversing destructive migration (`ADR-004/009`, NFR-010).

## 13. Security and Data Notes

- Application DB roles: migration owner, runtime read/write, and worker role where practical; runtime cannot alter schema.
- Parameterized queries only. Tenant-scoping integration tests cover every repository and API resource (AC-017).
- Password/session/reset token fields are write-only and excluded from serializer, support tooling, and logs.
- Conflict/change payloads contain private task content; encrypt storage via managed database encryption, restrict access, and apply retention jobs.
- `audit_events.metadata` uses a schema allowlist; arbitrary request bodies are forbidden.
- Backup inherits privacy classification and retention controls. Restore tests use restricted environments and sanitized verification where possible.
- Permanent account deletion includes users, profile, sessions, reset tokens, tasks/events, devices, commands, changes, conflicts, and jobs. Audit rows retain only non-identifying operational evidence through `ON DELETE SET NULL`.
- PWA logout must warn when pending commands exist; it must not silently delete DATA-014.
